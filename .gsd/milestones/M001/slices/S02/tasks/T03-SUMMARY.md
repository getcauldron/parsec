# T03: Build PyInstaller binary and verify identical sidecar behavior

**PyInstaller --onedir binary of the sidecar protocol handler builds, produces identical JSON output to unbundled Python, has correct stdout buffering, and works when spawned by Tauri.**

## What Happened

Created `sidecar_entry.py` as a thin PyInstaller entrypoint that forces line-buffered stdout via `io.TextIOWrapper` before any imports, sets environment variables to suppress PaddlePaddle C++ logging noise, then calls `sidecar.main()`. This separation keeps `sidecar.py` clean for testing.

Built `build_sidecar.sh` with `--onedir` mode, `--collect-all` for paddleocr/pyclipper/shapely and `--collect-data` for paddle/scipy, plus hidden imports for the full dependency tree. PyInstaller 6.19.0 on Python 3.13.

First build succeeded — 652MB total (41MB binary + 611MB `_internal/` with 152 files). Some warnings about missing `lmdb`/`skimage`/`imgaug` packages (not installed, referenced by PaddleOCR but not required for basic operation). No errors.

Verified the binary produces identical JSON to unbundled Python. Verified stdout is line-buffered (85ms response time, arrives before stdin close). Updated the Tauri sidecar wrapper to launch the PyInstaller binary with fallback to venv Python. Launched `cargo tauri dev`, confirmed sidecar spawned, clicked "Hello Sidecar" button, saw the JSON response in the UI.

## Deviations

- Used a wrapper script that exec's the PyInstaller binary rather than replacing the shell script entirely — this preserves fallback to venv Python when the binary hasn't been built, keeping `cargo tauri dev` working for all developers
- Didn't need to iterate on hidden imports as much as expected — first build worked for the hello/status protocol commands

## Known Issues

- Binary size is 652MB due to PaddlePaddle — acceptable for now but may need stripping for distribution (research flagged this as expected)
- PyInstaller warnings about `lmdb`, `skimage`, `imgaug` not being installed — these are optional PaddleOCR dependencies not needed for current functionality
- macOS code signing warnings for paddle's libblas.dylib and liblapack.dylib (SDK version 0,0,0) — will need addressing for production distribution
- Full OCR inference through the PyInstaller binary (PaddleOCR model loading, actual recognize()) not yet tested — that comes in S03 when process_file is wired
